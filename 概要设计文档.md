# 数据库内核智能研发运维平台架构设计文档

## 1. 文档概述

### 1.1 文档目的

本文档用于定义一个面向数据库内核团队的高度定制化 AI 平台的总体架构。该平台服务于数据库内核的**调研、设计、开发、测试、学习、运维与故障分析**全流程，核心目标是构建一个具备**静态理解、动态验证、运行观测、证据归因、知识沉淀**能力的闭环系统。

### 1.2 适用范围

本设计面向以下场景：

* PostgreSQL 及其衍生数据库内核研发
* 内核特性调研与方案设计
* Patch 开发与回归测试
* 逻辑复制、WAL、事务、快照、存储等机制研究
* 数据库运行态观测与性能归因
* 运维问题定位与标准化处理
* 团队知识资产建设与工程复用

### 1.3 设计目标

平台需要满足以下目标：

1. 支持源码、文档、社区资料的统一检索与理解
2. 支持数据库内核实验环境的快速构建与自动验证
3. 支持通过 eBPF 等手段进行运行态动态分析
4. 支持 Agent 驱动的复杂任务拆解与自动编排
5. 支持设计、开发、测试、运维过程中的证据链输出
6. 支持知识复用和团队经验沉淀
7. 支持安全可控的执行、审计与权限管理

---

## 2. 背景与问题定义

### 2.1 现状问题

传统数据库内核研发和运维面临以下典型问题：

1. **知识分散**
   源码、文档、邮件列表、提交历史、测试脚本、故障案例分散在不同位置，检索成本高，复用性差。

2. **静态分析与动态行为脱节**
   仅靠阅读源码可以理解设计，但难以确认实际运行路径、性能瓶颈和真实等待点。

3. **测试环境搭建复杂**
   数据库内核验证通常需要构建特定版本、特定拓扑、特定配置的环境，且往往需要多节点、多角色、多并发场景。

4. **问题归因依赖经验**
   很多问题需要综合源码、日志、性能数据、内核态行为才能定位，经验门槛高。

5. **知识沉淀不足**
   一次分析、一轮调试、一份 patch 评审、一次故障处置，常常难以沉淀为可复用资产。

### 2.2 建设目标

针对以上问题，需要建设一个统一平台，将下列能力组合起来：

* 静态代码与文档分析能力
* 实验环境创建与测试执行能力
* 运行态追踪与动态观测能力
* 跨阶段任务编排能力
* 结构化知识沉淀能力

---

## 3. 平台定位

平台定位为：

> 一个面向数据库内核全生命周期的智能工程平台，融合静态分析、容器化实验、自动化测试、eBPF 动态追踪、故障诊断与知识管理能力，支持数据库内核从调研到开发到验证到运维的全流程闭环。

它不是一个普通问答系统，而是一个**数据库内核智能实验与运维平台**。

## 4. 总体设计原则

### 4.1 证据驱动

平台输出不能仅基于语言模型“推测”，而应尽可能基于以下证据：

* 源码与调用链
* 测试执行结果
* 日志与指标
* trace 与 profile
* eBPF 运行态观测结果

### 4.2 静态与动态结合

平台需要同时支持：

* 静态理解：代码、文档、设计、结构、版本差异
* 动态验证：运行、测试、追踪、归因、回放

### 4.3 环境可复现

所有实验、测试、性能分析、故障演练都应支持环境模板化与复现。

### 4.4 模块解耦

知识层、Agent 层、执行层、观测层应解耦设计，避免强耦合导致扩展困难。

### 4.5 安全可控

所有写操作、环境操作、运维动作均应具备权限边界、风险隔离和审计能力。

### 4.6 面向数据库内核定制

平台不是通用研发平台，而是围绕数据库内核场景做深度定制，包括：

* 源码解析粒度
* 测试模板
* 实验拓扑
* eBPF 探针模板
* 故障模型
* 文档模板

## 5. 总体架构

## 5.1 分层架构

平台总体分为六层：

1. **交互接入层**
2. **智能编排层**
3. **静态分析层**
4. **实验执行层**
5. **动态观测层**
6. **知识沉淀层**

---

### 5.1.1 交互接入层

用于提供统一入口，面向不同角色提供不同交互方式。

组成：

* CLI
* Web 控制台
* IDE 插件
* Chat/IM 机器人
* API 接口

职责：

* 接收用户任务
* 展示分析结果、测试结果、trace 结果
* 提供环境管理、用例管理、报告查看能力

---

### 5.1.2 智能编排层

平台核心控制层，负责任务拆解、Agent 协同、工具调用与上下文管理。

组成：

* Task Planner
* Agent Orchestrator
* Context Manager
* Policy Engine
* Approval Engine
* Workflow Engine

职责：

* 将复杂需求拆解为多个步骤
* 路由到不同 Agent
* 管理多轮任务上下文
* 组织静态分析、环境构建、测试执行、trace 采集与归因
* 控制可执行动作的权限边界

---

### 5.1.3 静态分析层

负责对源码、设计文档、社区资料等进行静态理解和结构化分析。

组成：

* Source Indexer
* Symbol Analyzer
* Call Graph Analyzer
* AST/IR Parser
* Document Analyzer
* Diff/Patch Analyzer
* Version Comparison Engine

职责：

* 建立函数、结构体、宏、模块级索引
* 分析调用链和模块依赖
* 对 patch 做影响面分析
* 生成设计文档、测试方案和代码修改建议

---

### 5.1.4 实验执行层

负责搭建数据库实验环境并执行构建、启动、测试、故障注入等任务。

组成：

* Environment Manager
* Build Runner
* Cluster Orchestrator
* Test Runner
* Fault Injection Engine
* Artifact Collector

职责：

* 拉取代码
* 编译数据库
* 创建单机/主备/逻辑复制拓扑
* 执行 regression/isolation/TAP/perf 测试
* 收集日志、core、结果报告

---

### 5.1.5 动态观测层

负责运行时动态分析和性能归因。

组成：

* eBPF Probe Manager
* Uprobe/Kprobe Controller
* Trace Collector
* Perf Collector
* Runtime Event Bus
* Correlation Engine

职责：

* 对数据库关键进程和关键函数进行动态追踪
* 采集函数调用、耗时、等待、IO、网络、调度行为
* 将 trace 数据与测试任务、源码符号、日志进行关联
* 输出运行时证据链

---

### 5.1.6 知识沉淀层

负责统一存储研发运维过程中的知识与资产。

组成：

* Code Knowledge Base
* Document Knowledge Base
* Test Asset Library
* Trace Case Library
* Incident Library
* Design Archive
* Prompt/Template Library

职责：

* 存储源码索引、设计文档、测试资产、探针模板、分析报告
* 保存历史任务上下文、问题分析过程和解决结论
* 支持复用和检索

---

# 7.5 动态观测中心

### 7.5.1 功能定位

通过 eBPF/perf/trace 等手段获取数据库运行时真实行为。

### 7.5.2 核心设计

动态观测中心是平台的关键差异化能力之一，其目标不是简单采样，而是形成数据库语义级的运行证据。

### 7.5.3 观测对象

建议重点覆盖以下进程：

* postmaster
* postgres backend
* checkpointer
* bgwriter
* walwriter
* autovacuum worker
* walsender
* walreceiver
* logical replication launcher
* apply worker
* tablesync worker

### 7.5.4 观测维度

#### 1）用户态函数维度

通过 uprobes 跟踪关键函数：

* SQL 执行入口
* 事务开始与提交
* WAL 写入关键路径
* logical decoding 关键路径
* apply worker 关键路径
* utility command 关键路径

#### 2）内核态行为维度

通过 kprobes/tracepoints 跟踪：

* 文件 IO
* fsync
* 网络发送接收
* 调度切换
* block IO 延迟
* futex/latch/wait
* off-CPU 等待

#### 3）时序链路维度

构建事件级时序链：

SQL/复制消息
→ 用户态函数
→ WAL/解码/发送/应用
→ 内核态 IO/网络/调度
→ 完整闭环

### 7.5.5 探针模板化

eBPF 探针不应按临时脚本方式散落，而应沉淀为模板资产：

* SQL 生命周期模板
* WAL 路径模板
* Logical Decoding 模板
* Apply Worker 模板
* IO/FSYNC 模板
* Lock/Wait 模板
* Off-CPU 模板

### 7.5.6 结果组织

trace 结果应支持以下关联：

* 关联到测试任务
* 关联到环境实例
* 关联到 commit/patch
* 关联到源码函数
* 关联到具体告警/异常/日志

## 8. 关键业务流程设计

# 8.1 调研到开发流程

用户提出需求
→ Research Agent 检索资料
→ Static Analyzer 分析调用链与模块
→ Doc Agent 生成方案
→ Code Agent 生成 patch 草稿
→ 知识中心归档

---

# 8.2 开发到验证流程

提交 patch/设计
→ Environment Agent 创建实验环境
→ Build Manager 编译数据库
→ Test Agent 执行测试矩阵
→ Artifact Collector 收集结果
→ Diagnosis Agent 给出初步结论

---

# 8.3 动态观测流程

用户指定目标问题或测试任务
→ Runtime Trace Agent 选择探针模板
→ 动态观测中心加载 eBPF 探针
→ 执行测试任务
→ Trace Collector 采集事件
→ Correlation Engine 关联源码、测试、日志
→ 输出时序链与热点分析

---

# 8.4 运维诊断流程

接入日志/监控/现场信息
→ Ops Agent 做初步分类
→ 静态知识库匹配历史案例
→ 动态观测中心补充现场 trace
→ Diagnosis Agent 联合分析
→ 输出 runbook 与处置建议

---

## 9. 典型使用场景

### 9.1 场景一：验证逻辑复制 DDL patch

系统自动完成：

* 检索相关源码与历史 patch
* 分析 publication/subscription 涉及模块
* 构建 pub/sub 环境
* 编译 patch 分支
* 执行 DDL/DML 顺序验证
* 跟踪 walsender/apply worker
* 输出设计、测试、trace 结合报告

### 9.2 场景二：分析性能退化

系统自动完成：

* 拉基线版本与 patch 版本
* 运行基准测试
* 采集 perf 和 eBPF 数据
* 对比 TPS、延迟、WAL 量
* 定位热点函数与等待链路
* 输出性能回归报告

### 9.3 场景三：源码教学

系统自动完成：

* 检索目标函数
* 生成上下游调用链
* 提取关键结构体和事务路径
* 结合 trace 展示实际运行行为
* 输出教学材料

### 9.4 场景四：生产问题复盘

系统自动完成：

* 导入日志、配置、核心 SQL
* 建立近似环境
* 重放问题场景
* 加载观测模板
* 形成“现象-证据-根因-建议”报告

## 11. 技术选型建议

## 15. 风险与挑战

### 15.1 eBPF 使用复杂

* 权限要求高
* 不同内核环境兼容性复杂
* 探针设计不当可能引入额外开销

### 15.2 容器环境与真实环境存在差异

* 某些问题只能在 VM 或物理机复现
* 某些 IO/调度类问题在轻量容器中观察不充分

### 15.3 数据库版本差异大

* 不同版本函数名、结构体、执行路径不同
* 需要版本画像与兼容层

### 15.4 AI 结论需要证据约束

* 必须防止仅凭模型猜测输出高确定性结论
* 需要明确“证据充足/证据不足”的状态标识

### 15.5 系统复杂度高

* 平台本身涉及知识库、执行器、观测器、Agent、多种数据源整合
* 需要先聚焦 MVP，再逐步扩展

## 17. 总结

本平台的核心价值不在于“回答问题”，而在于构建一个面向数据库内核研发与运维的**证据驱动型智能工程闭环**。

它的真正能力体现在：

* 静态分析能解释设计与代码
* 实验环境能验证行为
* eBPF 能揭示运行真相
* Agent 能编排复杂任务
* 知识中心能沉淀长期资产

最终形成：

**调研 → 设计 → 开发 → 编译 → 测试 → 动态观测 → 归因 → 修复 → 沉淀**

这才是数据库内核团队真正可用的 AI 平台。

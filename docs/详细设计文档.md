
# 一、系统定义（最终定版）

> **DBK = Database Kernel Agent CLI**
> 一个以数据库工程流程为内核的多模型智能体终端系统

---

## 核心能力

DBK 默认支持：

* 需求分析（Requirements）
* 技术调研（Research）
* 架构设计（Design）
* 编码实现（Implement）
* 测试验证（Test）
* 运行时分析（Runtime Analysis）
* 文档生成（Doc）
* 运维与诊断（Ops / AIOps）

---

## 用户入口

```bash
dbk
```

进入后：

```text
DBK Mode Activated
provider: minimax
model: m2.7
repo: postgres
mode: db-engineering
```

## 5.5 数据库运行时指标字典（落地）

| 维度 | 指标 | 数据源 | 采样周期 | 告警阈值（建议初始值） |
| --- | --- | --- | --- | --- |
| Query | p50/p95/p99 latency | `pg_stat_statements` | 15s | p95 > 200ms 持续 5min |
| Wait | wait_event / wait_event_type | `pg_stat_activity` | 5s | `Lock` 占比 > 30% |
| IO | read/write latency, IOPS | `pg_stat_io` + eBPF | 15s | read latency > 10ms |
| Lock | blocked sessions, lock tree | `pg_locks` | 5s | blocked > 5 且持续 60s |
| Replication | replay/apply lag | `pg_stat_replication` | 5s | lag > 3s |
| Buffer | hit ratio | `pg_stat_database` | 60s | hit ratio < 95% |

## 5.6 Runtime Collector 设计

DBK runtime 统一把 SQL 采集、日志采集、系统采集转为标准事件：

```json
{
  "ts": "2026-01-01T00:00:00Z",
  "instance": "pg-main-01",
  "source": "pg_stat_activity",
  "category": "wait",
  "metric": "lock_wait_count",
  "value": 12,
  "labels": {
    "db": "postgres",
    "role": "primary"
  }
}
```

Collector 分层：

* SQL Collector：连接 DB 拉取运行时视图
* Log Collector：解析错误日志、慢查询日志
* Host Collector：CPU/内存/磁盘/网络（必要时接 eBPF）
* Trace Collector：接收 perf/eBPF/bpftrace 产物并归档

---

## 5.7 eBPF 可观测性方案（落地）

默认策略：**按需触发，不常驻**

触发条件（任一满足）：

* p95 latency 异常升高
* wait events 异常集中（如 `Lock` / `IO`）
* IO latency 与 SQL 指标不一致，需要内核侧证据

预置 trace profile：

* `cpu-hotpath`：`profile` + flamegraph，定位 CPU 热点
* `io-latency`：`biolatency` / `biosnoop`，定位块设备延迟
* `offcpu`：`offcputime`，定位调度等待
* `tcp-latency`：`tcplife` / `tcpconnect`，定位网络抖动
* `syscall-heavy`：`execsnoop` / `opensnoop`，定位异常系统调用

执行约束：

* 单次 trace 默认 30s，最长 120s
* 仅在用户审批后执行提权 trace
* 自动生成 `trace-summary.md`（Top N 栈、热点函数、结论）

---

## 5.8 权限与开销控制

权限模型：

* 普通模式：仅 SQL + logs + 基础系统指标
* 受限提权模式：允许 eBPF/perf，必须二次确认

开销预算：

* 常规观测开销目标 < 3%
* 短时 trace 峰值开销目标 < 8%
* 超预算自动降级（降采样 / 提前停止 trace）

---

## 5.9 运行时数据存储模型

推荐落地在 sqlite（MVP 阶段）：

```sql
CREATE TABLE runtime_metric (
  id INTEGER PRIMARY KEY,
  ts TEXT NOT NULL,
  instance TEXT NOT NULL,
  metric TEXT NOT NULL,
  value REAL NOT NULL,
  labels_json TEXT
);

CREATE TABLE trace_artifact (
  id INTEGER PRIMARY KEY,
  task_id TEXT NOT NULL,
  profile TEXT NOT NULL,
  started_at TEXT NOT NULL,
  duration_sec INTEGER NOT NULL,
  artifact_path TEXT NOT NULL,
  summary_json TEXT
);
```

归档路径：

```text
.dbk/artifacts/runtime/<task_id>/
  metrics/
  traces/
  flamegraphs/
  trace-summary.md
```

---

## 5.10 自动诊断与 RCA 证据链

RCA 统一按证据链组织：

```text
symptom
  -> metric anomaly
  -> wait/lock/plan evidence
  -> kernel evidence (eBPF/perf)
  -> root cause
  -> fix + rollback
```

输出要求：

* 每条结论必须绑定证据（SQL 结果、日志片段、trace 文件）
* 结论区分“高置信/中置信/低置信”
* 必须给出“下一步验证命令”

---

## 5.11 可验收标准（Phase 3 入口）

最小可用验收（MVP）：

* 能持续采集 6 类 DB runtime 指标并查询最近 24h
* 能在“数据库延迟升高”场景自动触发诊断流
* 能按审批执行至少 2 类 eBPF profile 并输出摘要
* 每次 RCA 产出可追溯的 evidence bundle
* 关键流程有回归测试（collector / parser / analyzer）
